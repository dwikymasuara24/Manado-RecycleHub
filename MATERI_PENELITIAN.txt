------------------------------------------------------------------------------
1. LATAR BELAKANG
------------------------------------------------------------------------------

Kota Manado sebagai ibukota Provinsi Sulawesi Utara menghadapi tantangan
serius dalam pengelolaan sampah perkotaan. Volume sampah yang terus meningkat
seiring pertumbuhan penduduk tidak diimbangi dengan sistem pengumpulan yang
efisien dan terorganisasi. Beberapa permasalahan utama yang teridentifikasi:

  a) Tidak adanya sistem digital untuk masyarakat mengajukan permintaan
     penjemputan sampah secara mandiri.
  b) Proses penjadwalan petugas lapangan masih dilakukan secara manual,
     rawan kesalahan, dan tidak efisien.
  c) Tidak ada mekanisme optimasi rute, sehingga petugas sering menempuh
     jarak yang tidak perlu (boros bahan bakar & waktu).
  d) Admin tidak memiliki dasbor real-time untuk memantau distribusi
     permintaan antar wilayah kecamatan.
  e) Belum ada integrasi peta digital (GIS) untuk visualisasi lokasi
     penjemputan dan rute perjalanan petugas.
  f) Partisipasi masyarakat dalam daur ulang masih rendah karena tidak
     ada platform informasi dan edukasi yang mudah diakses.

Manado Recycle Hub (MRH) dibangun untuk menjawab seluruh permasalahan
tersebut melalui platform web berbasis PHP/MySQL yang mengintegrasikan
algoritma penjadwalan cerdas (Priority Rule + Nearest Neighbor) dengan
sistem peta GIS berbasis OpenStreetMap/Leaflet.js.

2. LUARAN PENELITIAN
------------------------------------------------------------------------------

2.1 Luaran Utama
  - Sistem informasi manajemen sampah berbasis web (PHP, MySQL, Leaflet.js)
  - Modul algoritma penjadwalan: Priority Rule + Nearest Neighbor (Haversine)
  - Modul GIS: peta interaktif titik penjemputan + visualisasi rute petugas
  - Laporan harian, mingguan, dan bulanan otomatis berbasis data real

2.2 Luaran Pendukung
  - Fitur Daur Ulang (pickup_requests): warga ajukan penjemputan sampah
  - Fitur Clean Up Service (cleanup_requests): pembersihan lokasi tertentu
  - Panel Admin: verifikasi request, kelola petugas, generate jadwal
  - Panel Officer (Petugas): lihat jadwal, update status, live tracking
  - Blog & konten edukasi daur ulang (DIY Projects, Idea Box)
  - Kuesioner digital: pengumpulan data persepsi masyarakat
  - Analisis data pivot: berat sampah per kategori, per kecamatan
  - Notifikasi real-time via WhatsApp (trigger otomatis saat status berubah)
  - Progressive Web App (PWA): dapat diakses offline

5. VARIABEL PENELITIAN
------------------------------------------------------------------------------

5.1 Variabel Input (Masukan Algoritma)

  Dari pickup_requests / cleanup_requests:
    - kecamatan       : nama kecamatan lokasi penjemputan
    - latitude        : koordinat lintang lokasi (desimal, WGS84)
    - longitude       : koordinat bujur lokasi (desimal, WGS84)
    - status          : hanya request berstatus 'dikonfirmasi' yang diproses
    - created_at      : waktu pengajuan (untuk urutan FIFO dalam kecamatan)

  Dari officers:
    - id              : ID petugas aktif
    - status          : hanya 'aktif' yang diikutkan dalam penugasan

  Parameter Sistem:
    - DEPOT_LAT/LNG   : koordinat titik awal petugas (gudang MRH)
    - tanggal         : tanggal target penjemputan
    - tipe_layanan    : 'pickup' atau 'cleanup'
    - max_titik       : batas maksimal 30 titik per jadwal/hari

5.2 Variabel Proses (Dihitung oleh Algoritma)

  Priority Rule:
    - count_per_kecamatan : jumlah request dikonfirmasi per kecamatan
    - dist_to_depot       : jarak pusat kecamatan ke depot (tie-breaker)
    - rank                : urutan prioritas pelayanan kecamatan

  Nearest Neighbor (Haversine):
    - haversine_distance  : jarak Euclidean di permukaan bumi (km)
    - distance_from_prev  : jarak dari titik sebelumnya ke titik saat ini
    - total_route_km      : total jarak rute dalam satu jadwal

5.3 Variabel Output (Keluaran Sistem)

  - schedule_id      : ID jadwal yang dibuat
  - urutan_kunjungan : urutan titik yang akan dikunjungi petugas
  - officer_id       : petugas yang ditugaskan (round-robin)
  - tanggal_jadwal   : tanggal pelaksanaan penjemputan
  - total_titik      : jumlah lokasi dalam satu jadwal
  - rute_optimal     : urutan lokasi dengan total jarak minimum
  - status_request   : berubah dari 'dikonfirmasi' → 'dijadwalkan'


------------------------------------------------------------------------------
6. TAHAP ALGORITMA
------------------------------------------------------------------------------

6.1 Algoritma 1: Priority Rule (Penentuan Prioritas Kecamatan)

  TUJUAN: Menentukan kecamatan mana yang dilayani lebih dahulu
          berdasarkan volume permintaan yang menunggu.

  LANGKAH-LANGKAH:

  Langkah 1 — Kelompokkan Request per Kecamatan
    Input: semua request berstatus 'dikonfirmasi'
    Proses: group by kecamatan → hitung COUNT per kecamatan

  Langkah 2 — Pilih Kecamatan dengan Count Terbanyak (Greedy)
    Bandingkan count semua kecamatan.
    Pilih yang paling banyak sebagai "Best".

  Langkah 3 — Tie-Breaking (jika count sama):
    a) Hitung jarak pusat kecamatan ke Depot (Haversine)
    b) Kecamatan yang lebih dekat ke Depot diprioritaskan
    c) Jika jarak pun sama → urutan alfabetis (deterministik)

  Langkah 4 — Tandai Kecamatan Terpilih, Lanjut ke Sisa
    Request kecamatan terpilih dikeluarkan dari pool.
    Ulangi Langkah 1-3 untuk request yang tersisa.
    Berhenti ketika semua request sudah terurut.

  OUTPUT: Urutan kecamatan [ Kec_A, Kec_B, Kec_C, ... ]

  CONTOH:
    Pool: Wenang=8, Tikala=3, Sario=1
    Putaran 1 → pilih Wenang (8) → hapus dari pool
    Putaran 2 → pilih Tikala (3) → hapus dari pool
    Putaran 3 → pilih Sario  (1) → selesai
    Hasil: [ Wenang → Tikala → Sario ]


6.2 Algoritma 2: Nearest Neighbor + Haversine (Optimasi Rute)

  TUJUAN: Menentukan urutan kunjungan titik-titik penjemputan
          dalam satu kecamatan agar total jarak minimum.

  RUMUS HAVERSINE (jarak antara 2 koordinat di bumi):
    a = sin²(Δlat/2) + cos(lat1) × cos(lat2) × sin²(Δlon/2)
    c = 2 × atan2(√a, √(1−a))
    d = R × c    (R = 6371 km, jari-jari bumi)

  LANGKAH-LANGKAH:

  Langkah 1 — Inisialisasi
    titik_saat_ini = DEPOT
    unvisited = semua titik request dalam kecamatan
    visited = []

  Langkah 2 — Temukan Tetangga Terdekat
    Hitung Haversine(titik_saat_ini, setiap titik di unvisited)
    Pilih titik dengan jarak MINIMUM → nearest

  Langkah 3 — Kunjungi Titik Terdekat
    Tambahkan nearest ke visited
    Catat jarak dari titik sebelumnya (distance_from_prev)
    titik_saat_ini = nearest
    Hapus nearest dari unvisited

  Langkah 4 — Ulangi
    Kembali ke Langkah 2 sampai unvisited kosong

  Langkah 5 — Kembali ke Depot
    Hitung jarak dari titik terakhir ke DEPOT
    Tambahkan sebagai titik penutup rute

  OUTPUT: [ DEPOT → T3 → T1 → T4 → T2 → DEPOT ]
          Dengan total jarak = Σ semua distance_from_prev

  CONTOH (4 titik):
    DEPOT → A (2 km) → B (3 km) → C (4 km) → D (3 km) → DEPOT (4 km)
    Total = 16 km


6.3 Kombinasi Kedua Algoritma (generateSchedule)

  ALUR LENGKAP:
    1. Ambil semua request status='dikonfirmasi' dari DB
    2. Jalankan Priority Rule → urutan kecamatan
    3. Ambil kecamatan prioritas #1 beserta request-nya
    4. Jalankan Nearest Neighbor pada request kecamatan tsb
    5. Simpan ke tabel: schedules, routes, schedule_requests
    6. Update status request → 'dijadwalkan'
    7. Assign petugas (round-robin dari daftar officer aktif)
    8. Hitung tanggal jadwal = tanggal_mulai + N hari
    9. Kirim notifikasi WhatsApp otomatis
   10. Hapus request yang sudah dijadwalkan dari pool
   11. Ulangi dari Langkah 2 sampai pool habis

  BATASAN SISTEM:
    - Maks. 30 titik per jadwal (1 hari per petugas)
    - Proteksi: tidak boleh ada 2 jadwal kecamatan sama pada hari sama
    - Petugas ditugaskan secara round-robin (fair distribution)


------------------------------------------------------------------------------
7. INTEGRASI GIS (Geographic Information System)
------------------------------------------------------------------------------

7.1 Teknologi GIS yang Digunakan

  Library   : Leaflet.js v1.9.4 (open-source)
  Tile Map  : OpenStreetMap (OSM) — bebas biaya, tanpa API key
  Koordinat : WGS84 (desimal, latitude/longitude)
  Formula   : Haversine (diimplementasikan di PHP & JavaScript)

7.2 Fitur GIS dalam Sistem

  a) Peta Input Request (daur_ulang.php / cleanup_service.php)
     - Warga klik peta → koordinat otomatis terisi ke form
     - Reverse geocoding → nama kelurahan & kecamatan terisi otomatis
     - Marker dapat digeser untuk koreksi lokasi
     - Fallback: warga isi alamat teks jika tidak mau klik peta

  b) Visualisasi Rute Petugas (admin/rute_jadwal.php)
     - Peta menampilkan semua titik request dalam kecamatan terpilih
     - Warna marker: biru (Daur Ulang), oranye (Clean Up), hijau (Depot)
     - Setelah algoritma NN dijalankan → polyline rute ditarik di peta
     - Label nomor urut kunjungan tampil di setiap marker
     - Popup klik marker: kode request, nama pemohon, jenis wadah

  c) Live Tracking Petugas (admin/live_tracking.php)
     - Monitoring posisi petugas saat bertugas di lapangan
     - Peta menunjukkan pergerakan petugas secara real-time

  d) Peta Lokasi Depot (lokasi_kami.php)
     - Menampilkan lokasi depot/gudang MRH kepada publik

7.3 Cara Kerja Integrasi Koordinat

  SAAT WARGA SUBMIT REQUEST:
    1. Warga buka halaman daur_ulang.php / cleanup_service.php
    2. Peta Leaflet muncul — warga klik titik lokasi penjemputan
    3. JavaScript tangkap event click → ambil lat/lng
    4. Reverse geocoding (Nominatim OSM API) → isi kelurahan, kecamatan
    5. Lat/lng disimpan ke hidden input → dikirim bersama form
    6. Server simpan ke kolom latitude, longitude di tabel pickup_requests

  SAAT ADMIN GENERATE JADWAL:
    1. Query semua request dikonfirmasi (dengan lat/lng)
    2. Jika lat/lng kosong → gunakan koordinat pusat kecamatan + random jitter
    3. Algoritma Haversine hitung jarak antar titik
    4. Hasil rute dikirim ke frontend sebagai JSON
    5. JavaScript Leaflet render polyline di peta

7.4 Struktur Data GIS dalam Database

  Tabel pickup_requests:
    latitude   DECIMAL(10,8)  — koordinat lintang
    longitude  DECIMAL(11,8)  — koordinat bujur

  Tabel routes:
    schedule_id        — referensi jadwal
    urutan             — urutan kunjungan (0 = depot, 1,2,3... = titik)
    pickup_request_id  — referensi titik request
    dist_from_prev_km  — jarak dari titik sebelumnya (km, Haversine)

  Tabel kecamatan:
    nama_kecamatan     — nama kecamatan
    lat, lng           — koordinat pusat kecamatan (referensi algoritma)

8.2 Alur Status Request

  [menunggu] → [dikonfirmasi] → [dijadwalkan] → [sedang_diproses] → [selesai]
                    ↑                  ↑
                  ADMIN            ALGORITMA
               (manual cek)     (otomatis assign)

  Catatan: 'dibatalkan' bisa terjadi dari status manapun oleh Admin.

8.3 Ringkasan Peran Setiap Aktor

  WARGA    : Ajukan request → terima notifikasi status
  ADMIN    : Verifikasi request → generate jadwal → pantau dashboard
  ALGORITMA: Priority Rule (urutan kecamatan) + NN (rute titik)
  PETUGAS  : Ikuti rute → update status → catat berat sampah
  SISTEM   : Kirim notifikasi WA → simpan log → buat laporan otomatis
